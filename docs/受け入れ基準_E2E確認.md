# 受け入れ基準と E2E 確認項目（最終版）

## 1. 受け入れ基準

### クライアント側

| # | 基準 | 確認方法 |
|---|------|----------|
| 1 | 匿名ユーザーでも主要 UIUX の確認が可能 | 匿名ログインで起動し、録音〜提出導線まで操作可能であること |
| 2 | 「練習」と「提出済み」を明確に区別して閲覧できる | 録音履歴画面でタブ切り替えができ、ステータスが正しく表示されること |
| 3 | 提出前確認モーダルで共有相手・取り消し不可を明示 | 先生に送る押下時にモーダルが表示され、内容が適切であること |
| 4 | 提出後、成功トーストと提出済みタブへの遷移が行われる | 提出後にトーストが表示され、録音履歴の提出済みタブに遷移すること |
| 5 | 画面上部に「提出済みのみコンサルに共有」の説明が常時表示される | 録音履歴画面の ShareBoundaryNotice が表示されていること |

### ハンバーガーメニュー

| # | 基準 | 確認方法 |
|---|------|----------|
| 6 | メニュー項目が AuthStage / ロールで正しく出し分けされる | 匿名時・ログイン時・コンサル時・管理者時で表示項目が異なること |
| 7 | 匿名時は「データ保存について」「アカウント作成」が表示される | 匿名状態でドロワーを開き、該当項目があること |
| 8 | ログイン時は「録音履歴」「担当コンサルへ連絡」が表示される | ログイン状態でドロワーを開き、該当項目があること |
| 9 | コンサル時は「講師用ダッシュボード」が表示される | consultant_assignments に登録されたユーザーまたは開発トグル ON で表示 |
| 10 | 管理者時は「管理者ダッシュボード」が表示される | 開発トグル ON または app_role=admin で表示 |

### コンサルタント側

| # | 基準 | 確認方法 |

## 2. E2E 確認フロー（推奨順）

### Step 1: 匿名ログインでの UI 確認

1. アプリ起動（匿名サインイン）
2. ハンバーガーメニューを開く → 「データ保存について」「アカウント作成」を確認
3. 「開発: ログイン済み画面」を ON
4. メニューを再開 → 「録音履歴」「講師用」「管理者」の表示有無を確認（開発トグルで切り替え）
5. 録音履歴画面を開く → ShareBoundaryNotice を確認
6. 会話トレーニングで録音 → 練習保存 → 「先生に送る」→ 確認モーダル → 送信 → 録音履歴（提出済み）への遷移を確認

### Step 2: コンサルタント画面の確認

1. メニューで「開発: コンサルタント」を ON、または consultant_assignments にテストデータを投入
2. 講師用ダッシュボードを開く
3. KPI・提出キュー・詳細ボタン・課題発行タブを確認
4. 提出カードで再生・テンプレ挿入・フィードバック送信を実行

### Step 3: 管理者画面の確認

1. メニューで「開発: 管理者」を ON
2. 管理者ダッシュボードを開く
3. 権限・監査・運用・AI承認の 4 タブを確認
4. 運用タブで未対応件数が表示されることを確認

### Step 4: Phase A マイグレーション適用後

1. Phase A の 7 本の SQL を適用
2. 上記 Step 1〜3 を再実行し、エラーが出ないことを確認

### Step 5: Phase B 適用後（本番前）

1. JWT に app_role=admin を設定
2. Phase B の SQL を適用
3. クライアント・コンサル・管理者の権限が期待どおりに分離されていることを確認
