# Phase C: 迷いナッジ（PLANモード用プロンプト）

## 実装PLAN

- **phaseC_impl.md** に実装PLANを記載済み。着手時はそちらを参照する。

## 使い方

- Cursor の PLAN モードを開く
- 下の「Prompt」をそのまま貼る
- 返ってきた PLAN をもとに、まず KPI 計測を仕込んでから UI ナッジを実装する

## Prompt（そのまま貼り付け）

```text
Engrowth アプリの UI/UX 強化として、Phase C「迷いナッジ」を実装したいです。
ユーザーが迷わず次アクションへ進めるように、ポップアップ・ボトムシート・控えめな誘導アニメーションを使った実装 PLAN を作成してください。
MASTER_PLAN.md の Sound-First / Zero-Latency / Consultant Bridge / Guardrails に従ってください。

【目的】
- ユーザーの迷い時間を減らし、主要CTAへの到達率を上げる
- 誘導を「邪魔な割り込み」ではなく「優しい後押し」に統一する
- 起動時、学習開始前、未操作時の導線設計を全体最適する

【前提】
- 起動時1日1回のショートカットポップアップは実装済み
- 学習開始前の LearningIntroDialog は一部導線で実装済み
- analytics イベント送信基盤は既存利用可能

【必須要件】
1) PLAN は以下の章立てで作成する
   - Goal
   - Non-goal
   - Exit Criteria
   - 影響ファイル一覧
   - ナッジパターン設計（表示条件/優先順位/非表示条件）
   - 実装ステップ（1PR=1目的）
   - 計測イベント設計
   - QAシナリオ

2) ナッジ設計を3層で整理する
   - 起動時ナッジ（既存 popup の最適化）
   - コンテキストナッジ（学習開始前の説明や次行動提案）
   - 迷い検知ナッジ（例: 3秒無操作時にCTAを軽く強調）

3) UI手段の使い分けを定義する
   - showModalBottomSheet を使う条件
   - ダイアログを使う条件
   - インライン誘導（バッジ/微バウンド）を使う条件

4) 計測を先に決める
   - nudge_shown / nudge_dismissed / nudge_accepted
   - 迷い時間（初回表示→CTAタップまで）
   - ナッジ有無での到達率比較

5) 失敗パターン回避を明記する
   - 出しすぎによる疲労
   - 読みづらい文言
   - 同一セッションでの過剰再表示

【制約】
- 1PR=1目的で段階導入
- 既存の 1日1回ルールや導線を尊重
- テーマ準拠（engrowth_theme.dart）
- 体験テンポを崩さない（Zero-Latency）

【出力形式】
- PR単位で「対象ファイル」「表示条件」「文言案」「受け入れ条件」「計測イベント」を提示
- 最初に着手すべき最小構成（MVP）と、その次の拡張案まで示す
```
