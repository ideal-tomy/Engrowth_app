# ロール別受け入れ基準・E2E

## Goal

- ロール別（ユーザー/コンサル/管理者）の受け入れ基準を整備する。
- E2E チェックリストを整備し、リリース前検証に活用する。

## Non-Goal

- 自動E2Eテストの実装。チェックリストと手順の文書化に留める。

## Exit Criteria

- 各ロールの受け入れ基準が文書化され、実装完了判定に使える。
- E2E チェックリストが実行可能な手順として整備されている。

---

## 1. ユーザー（学習者）受け入れ基準

| # | 基準 | 確認方法 |
|---|------|----------|
| U1 | 匿名ユーザーでも主要 UI の確認が可能 | 匿名ログインで録音〜提出導線まで操作可能 |
| U2 | 練習/提出済みを明確に区別して閲覧できる | 録音履歴のタブ切り替え、ステータス表示 |
| U3 | 提出前確認モーダルで共有相手・取り消し不可を明示 | 先生に送る押下時にモーダル表示 |
| U4 | 提出後に成功トーストと提出済みタブへの遷移 | 提出後のフロー確認 |
| U5 | 提出済みのみコンサルに共有の説明が常時表示 | ShareBoundaryNotice 表示 |
| U6 | 初回導線から迷子にならずホームに戻れる | チュートリアル完了後の遷移 |
| U7 | 60〜90秒で聞く→話す→返答を完了可能 | チュートリアル所要時間 |

---

## 2. コンサルタント受け入れ基準

| # | 基準 | 確認方法 |
|---|------|----------|
| C1 | 提出レビュー高速化が行える | 再生・テンプレ・コメント・ステータス更新 |
| C2 | 詳細ログの深掘りが行える | 詳細ボタンでドロワー表示（実データ or プレースホルダー） |
| C3 | KPI（未対応・本日対応・担当数）が表示される | 提出キュー上部の KPI カード |
| C4 | 課題発行タブが存在する | タブ切り替えで表示（プレースホルダー含む） |
| C5 | 担当外クライアントの提出を読めない | RLS 検証（Phase B 適用後） |

---

## 3. 管理者受け入れ基準

| # | 基準 | 確認方法 |
|---|------|----------|
| A1 | 4タブ（権限付与/監査/運用/AI承認）が表示される | 管理者ダッシュボード |
| A2 | 運用監視で未対応提出件数が取得できる | 運用タブ表示 |
| A3 | 管理者のみ全件閲覧可能 | app_role=admin で検証（Phase B） |

---

## 4. 共通・セキュリティ

| # | 基準 | 確認方法 |
|---|------|----------|
| S1 | Phase B 適用後、クライアントが他人の提出を読めない | 直接取得で空となること |
| S2 | Phase B 適用後、コンサルが未担当を読めない | 一覧に表示されないこと |
| S3 | ライト/ダーク双方で可読性・UI統一 | テーマ切替後も colorScheme 準拠 |

## 5. E2E チェックリスト（推奨順）

### Step 1: 匿名ログインでの UI 確認

1. アプリ起動（匿名サインイン）
2. ドロワーを開く → データ保存について・アカウント作成を確認
3. 開発トグル ON でメニュー出し分け確認
4. 録音履歴 → ShareBoundaryNotice 確認
5. 会話トレーニング → 録音 → 練習保存 → 先生に送る → 確認モーダル → 送信 → 提出済みタブ遷移

### Step 2: コンサルタント画面

1. devViewAsConsultant ON または consultant_assignments 投入
2. 講師用ダッシュボードを開く
3. KPI・提出キュー・詳細・課題発行タブを確認
4. 提出カードで再生・テンプレ挿入・フィードバック送信

### Step 3: 管理者画面

1. devViewAsAdmin ON
2. 管理者ダッシュボードを開く
3. 4タブ確認、運用タブで未対応件数確認

### Step 4: チュートリアル導線

1. 新規 or リセットでオンボーディング開始
2. オンボーディング完了 → チュートリアル → ホーム戻り
3. 所要時間・離脱ポイントの記録
