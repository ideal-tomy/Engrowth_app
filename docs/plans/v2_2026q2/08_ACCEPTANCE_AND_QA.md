# ロール別受け入れ基準・E2E

## Goal

- ロール別（ユーザー/コンサル/管理者）の受け入れ基準を整備する。
- E2E チェックリストを整備し、リリース前検証に活用する。

## Non-Goal

- 自動E2Eテストの実装。チェックリストと手順の文書化に留める。

## Exit Criteria

- 各ロールの受け入れ基準が文書化され、実装完了判定に使える。
- E2E チェックリストが実行可能な手順として整備されている。

---

## 1. ユーザー（学習者）受け入れ基準

| # | 基準 | 確認方法 |
|---|------|----------|
| U1 | 匿名ユーザーでも主要 UI の確認が可能 | 匿名ログインで録音〜提出導線まで操作可能 |
| U2 | 練習/提出済みを明確に区別して閲覧できる | 録音履歴のタブ切り替え、ステータス表示 |
| U3 | 提出前確認モーダルで共有相手・取り消し不可を明示 | 先生に送る押下時にモーダル表示 |
| U4 | 提出後に成功トーストと提出済みタブへの遷移 | 提出後のフロー確認 |
| U5 | 提出済みのみコンサルに共有の説明が常時表示 | ShareBoundaryNotice 表示 |
| U6 | 初回導線から迷子にならずホームに戻れる | チュートリアル完了後の遷移 |
| U7 | 60〜90秒で聞く→話す→返答を完了可能 | チュートリアル所要時間 |
| U8 | 担当コンサルへ連絡画面から報告・相談ができる | ドロワー「担当コンサルタントへ連絡」→連絡画面→クイック報告送信 | B15 |
| U9 | ResumeLearningCard 1タップで学習再開できる | 続きから/推奨/フォールバックで適切に遷移 | B16 |

## 2. コンサルタント受け入れ基準

| # | 基準 | 確認方法 |
|---|------|----------|
| C1 | 提出レビュー高速化が行える | 再生・テンプレ・コメント・ステータス更新 |
| C2 | 詳細ログの深掘りが行える | 詳細ボタンでドロワー表示。session_uuid 連携時は実データ（端末・セッション・7日傾向）、未連携時は理由表示 |
| C3 | KPI（未対応・本日対応・担当数）が表示される | 提出キュー上部の KPI カード |
| C4 | 課題発行が行える | 日課運用タブで担当選択・プリセット・カスタム文言・発行が可能。発行後プレビュー表示 |
| C5 | 担当外クライアントの提出を読めない | RLS 検証（Phase B 適用後） |

## 3. 管理者受け入れ基準

| # | 基準 | 確認方法 |
|---|------|----------|
| A1 | 4タブ（権限付与/監査/運用/AI承認）が表示される | 管理者ダッシュボード |
| A2 | 運用監視で未対応提出件数・課題発行状況が取得できる | 運用タブで未対応件数・本日の課題発行数・未発行担当数を表示 |
| A3 | 管理者のみ全件閲覧可能 | app_role=admin で検証（Phase B） |
| A4 | 権限付与タブで実データ CRUD が行える | consultant_assignments の追加・取り消し。監査ログに記録 | B13 |
| A5 | 監査タブで access_audit_logs 実データ表示・フィルタが行える | 一覧・権限のみフィルタ | B14 |
| A6 | 運用タブで配信デモ（LINE/LINE WORKS）の状態表示が行える | 課題＋チャネル別状態（in_app/LINE/LINE WORKS） | デモ |

## 4. 共通・セキュリティ

| # | 基準 | 確認方法 |
|---|------|----------|
| S1 | Phase B 適用後、クライアントが他人の提出を読めない | 直接取得で空となること |
| S2 | Phase B 適用後、コンサルが未担当を読めない | 一覧に表示されないこと |

## 5. E2E チェックリスト（推奨順）

### Step 1: 匿名ログインでの UI 確認

1. アプリ起動（匿名サインイン）
2. ドロワーを開く → データ保存について・アカウント作成を確認
3. 開発トグル ON でメニュー出し分け確認
4. 録音履歴 → ShareBoundaryNotice 確認
5. 会話トレーニング → 録音 → 練習保存 → 先生に送る → 確認モーダル → 送信 → 提出済みタブ遷移

### Step 2: コンサルタント画面

1. devViewAsConsultant ON または consultant_assignments 投入
2. 講師用ダッシュボードを開く
3. KPI・提出キュー・詳細・課題発行タブを確認
4. 提出カードで再生・テンプレ挿入・フィードバック送信
5. 詳細ボタンでドロワーを開き、実データ or 欠損理由を確認（B10）
6. 日課運用タブで担当クライアント選択・プリセット選択・課題発行（B11）
7. 発行後プレビューカード表示を確認

### Step 3: 管理者画面

1. devViewAsAdmin ON または app_role=admin（User Metadata）
2. 管理者ダッシュボードを開く
3. 4タブ確認（権限付与/監査/運用/AI承認）
4. 権限付与タブ: 担当割当一覧・権限追加（consultant_id, client_id）・取り消し
5. 監査タブ: access_audit_logs 実データ一覧・「権限のみ」フィルタ
6. 運用タブ: 未対応件数・本日課題発行数・配信デモ（in_app/LINE/LINE WORKS 状態）

### Step 4: チュートリアル導線

1. 新規 or リセットでオンボーディング開始
2. オンボーディング完了 → チュートリアル → ホーム戻り
3. 所要時間・離脱ポイントの記録

### Step 5: B15 担当コンサル連絡

1. ログイン済みで consultant_assignments に割当あり
2. ドロワーを開く → 「担当コンサルタントへ連絡」をタップ
3. 連絡画面（担当者情報・チャネル選択・クイック報告）が表示される
4. 報告種別を選択しメッセージ入力 → 送信 → 成功トースト
5. 担当未割当時は「担当コンサルタントが割り当てられていません」が表示される

### Step 6: B16 ResumeLearningCard 1タップ起動

1. ホームに ResumeLearningCard を表示
2. 続きあり状態でタップ → 学習画面が期待した文から開始される
3. 続きなし・推奨ありでタップ → 推奨文から開始される
4. アプリ再起動直後タップ → 読み込み完了後に正しい再開先へ遷移する（誤判定なし）
