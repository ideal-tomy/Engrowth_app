# AI 駆動・Supabase DB 由来の Flutter 音声再生システム — 作業履歴と結論レポート

**作成日**: 2026年3月  
**結論**: 本プロジェクトにおいて、Supabase データベースを唯一の音源とする Flutter 音声再生システムを、安定して成功させることはできなかった。  
**方針**: 本方式は打ち切りとし、本レポートでこれまでの作業・試行・原因を記録する。

---

## 1. 目指したこと

- **アプリ**: Flutter（Web / モバイル）
- **音声**: OpenAI TTS で生成した MP3 を Supabase Storage に保存し、メタデータを `tts_assets` テーブルで管理
- **再生**: アプリは DB の `cache_key`（テキスト・言語・voice・speed・モデルのハッシュ）で URL を引いて再生
- **事前投入**: `prefill_tts_assets.dart` で会話発話などを一括生成し、DB に「パッチ」として投入
- **課金方針**: 再生時は DB のみ参照し、ミス時は Edge/OpenAI を呼ばない（課金防止）

---

## 2. ハッシュ一致のために行ったこと

### 2.1 キャッシュキーの仕様の統一

- **キー文字列**: `正規化テキスト|language|voice|speed|model`
- **正規化**: trim、改行を `\n` に統一、小文字化（アプリ・prefill・Edge で同一ルール）
- **voice**: 話者 A → `alloy`、B → `nova`（`tts_voice_config.dart` / prefill の `_voiceForRole` / Edge の body）
- **speed**: 1.0 のとき `"1"`、0.6 のとき `"0.6"` など、浮動小数点の表記を Edge（JS）と Dart で揃えた
- **model**: `tts-1` に統一（`tts_cache_key_util.dart` の `kTtsModel`、Edge の `MODEL`、prefill のコメント・Edge 呼び出し）

### 2.2 モデル変更（tts-1-hd → tts-1）に伴う対応

- キャッシュキーにモデル名が含まれるため、モデル変更で既存 DB のキーがすべて「別キー」になることを明文化
- アプリ・Edge・prefill の 3 か所をすべて `tts-1` に揃え、prefill をやり直す手順をドキュメント化（`TTS_tts-1で音声を用意し直す手順.md` 等）

### 2.3 検証・照合スクリプト

- **verify_tts_cache_hash.dart**: `--text "..."` を正しく解釈するよう修正。指定テキストでキーを計算し、`tts_assets` に存在するか確認。話者（voice）が A/B で変わることを注記
- **check_tts_prefill_status.dart**: 全発話について「アプリと同じキーで DB に存在するか」を照合。  
  - Supabase のデフォルト 1000 件制限を避けるため、`conversation_utterances` を **ページネーション（.range）** で全件取得  
  - `in_('cache_key', batch)` は Dart クライアントで存在しないため **inFilter** に変更  
  - `.order()` のあとでは `.eq()` が使えないため、**filter を先に、order/limit を後に** するようクエリ順を変更（`query` を dynamic で受ける）
- **report.csv**: 発話ごとに `in_tts_assets: YES/NO` を出力。証拠・切り分け用

### 2.4 prefill のデータソースとページネーション

- **conversation_utterances のみ・先頭 1000 件のみ** だと、後ろの会話や「3分会話」が漏れる可能性を指摘
- prefill を **conversation_utterances を .range でページネーションして全件** 取得するよう変更
- **sentences** テーブル（シナリオ学習・即興作文など）も TTS 対象として prefill に追加
- 重複排除は `text|language|voice|speed` の Set で実施（model は Edge 側で付与）

### 2.5 ドキュメント整備（ハッシュ合わせ・確認手順）

- **TTS_キャッシュを合わせる方法_やさしい版.md**: モデル・voice・正規化を 3 か所で揃える手順
- **TTS_音声DB確認手順_やさしい版.md**: report.csv の出し方と YES/NO の見方
- **TTS_tts-1で音声を用意し直す手順.md**: tts-1 で一から揃える 3 ステップ

## 3. 発生したエラーと対応

### 3.1 TTS cache miss（音声が DB にない）

- **現象**: 特定発話で「TTS cache miss: 音声がDBにありません。prefill を実行してください。」とレシピ生文字列がログに出る
- **対応**:  
  - 該当テキストが `conversation_utterances` か `sentences` に含まれるか確認  
  - prefill のページネーション・sentences 取り込みで「取りこぼし」を減らす  
  - 再生ループでは **Object catch** でキャッシュミス系メッセージを捕捉し、**当該発話をスキップして continue**。スキップ前に **`_ttsService.stop()`** を呼び、次の発話で状態がおかしくならないようにした
- **結果**: キャッシュミス自体は「スキップして続行」されるが、**一度でもミスが発生すると、以降はプログレスバーだけ進み音声が鳴らない** という別現象が残った

### 3.2 最初の 1 フレーズしか再生されない

- **現象**: 「3分一気に聴く」で 1 本目だけ再生され、2 本目以降が鳴らない。report.csv では多数が YES
- **想定原因**:  
  - **Web オートプレイ制限**: 2 本目以降の `play()` がユーザー操作とみなされずブロックされ、`TtsPlaybackBlockedException` でループ break  
  - **再生速度の不一致**: アプリが 0.6 で検索しているのに prefill が 1.0 のみで、2 本目以降が DB ミス
- **対応**:  
  - Web でブロックされたときに「続きを再生」SnackBar を表示し、ユーザーがタップしたら `startIndex` を指定して同じ再生ループを再開  
  - 再生速度 1.0 での確認と、prefill の `--include-slow`（0.6 投入）を案内
- **結果**: ユーザー環境では「続きを再生」は表示されず、1.0 で再生しても **1 回でもエラー（キャッシュミス）が出ると、以降はプログレスバーだけ進み音声が鳴らない** 状態が継続

### 3.3 1 回エラー後にすべてプログレスバーだけ進む

- **現象**: いったん TTS cache miss などが 1 回出ると、その後はプログレスバーだけ進み、音声は再生されない
- **対応**:  
  - キャッシュミスを **Object catch** で確実に捕捉  
  - スキップ前に **`await _ttsService.stop()`** で再生状態をリセット  
  - 例外を「未処理の Promise」にしないよう、キャッチ範囲を広げた
- **結果**: **改善せず**。根本原因（再生パイプラインや Web の状態が 1 回の例外で破綻する等）は特定できず

### 3.4 setState() called after dispose()

- **現象**: 会話再生中に画面を戻るなどで dispose されたあと、非同期で setState が呼ばれエラー。その後、別会話をタップしても再生されない
- **対応**: `_playAllConversation` および `_playUtterance` 内で、**await の直後や setState の前に `if (!mounted) break;` / `if (!mounted) return;`** を追加。`recordPlayback` 後の setState は `if (mounted && _mode == 'listen')` でガード
- **結果**: dispose 後の setState は防げるが、「1 回エラー後に鳴らなくなる」事象とは別要因と判断

### 3.5 その他のエラー（参考）

- **429 insufficient_quota**: OpenAI のクォータ超過。verify や Edge 呼び出し時に発生。課金・クレジットの確認が必要
- **increment_daily_done 404**: Supabase に該当 RPC が存在しない。アプリ側のフォールバックで動作する可能性あり
- **achievements テーブル不在 (PGRST205)**: マイグレーション未適用等。TTS 再生の成否とは別

## 4. Edge フォールバックの試行と取りやめ

- **試行**: DB ミス時に「課金防止」を緩め、**Edge `tts_synthesize` を呼んで生成・保存・返却 URL で再生**するように変更。prefill が未完了でも鳴るようにした
- **取りやめ**: **再生のたびに OpenAI が呼ばれると課金が止まらなくなる** ため、**DB のみ参照し、ミス時は例外をスローする元の仕様に戻した**

## 6. 成功させられなかった主な原因の整理

1. **再生パイプラインの脆弱性**  
   1 回でも TTS 例外（キャッシュミス等）が発生すると、その後の発話が再生されず、プログレスバーだけ進む。Web の Audio 状態、Flutter の再生状態、またはその組み合わせのどこかが 1 回の例外で破綻している可能性が高いが、特定できなかった。

2. **Web オートプレイ・環境依存**  
   「続きを再生」で 2 本目以降を鳴らす対策を入れたが、再現環境では効果が確認できず。ブラウザやユーザー操作の違いに強く依存している可能性がある。

3. **DB とアプリのキー一致だけでは足りない**  
   照合スクリプトと report.csv では「同じキーで DB に存在する」と判断できる発話が多数あるにもかかわらず、アプリでは再生されない。キー一致は必要条件だが、**再生の安定性を保証する十分条件にはなっていない**。

4. **課金制約**  
   DB ミス時に Edge で都度生成すると、運用コストが許容できないため、**DB のみで完結させる前提**だった。その前提の下では、prefill の取りこぼしや 1 回エラー後の不具合が解消できず、システムとして成立しなかった。

---

## 7. 結論

- **AI 駆動による Supabase データベースからの Flutter 音声再生システムを、現仕様（DB のみ・課金抑制）のまま安定して実現することは、本作業の範囲では不可能と判断した。**
- ハッシュ一致のための施策（モデル・正規化・voice・speed の統一、prefill の拡充、照合スクリプトの整備）は実施したが、**「report 上はほぼ完了しているのに、アプリでは 1 本目だけ／1 回エラー後に全く鳴らない」** という事象を解消できなかった。
- DB ミス時の Edge フォールバックは、課金が継続するため採用せず、**DB のみの仕様に戻して打ち切り**とした。

今後の代替案の検討（例: デバイス TTS のみ、または別キャッシュ方式の検討）は、本レポートの前提・経緯を参照して行うことを推奨する。
