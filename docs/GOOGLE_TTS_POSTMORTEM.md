# Google TTS ポストモーテム

## 概要

Google Cloud Text-to-Speech API（Wavenet）を採用し、高品質な英語発音を実現する方針で実装を進めたが、Web/モバイルブラウザにおける自動再生制約により、設計目標を満たすことができなかった。本ドキュメントはその原因と再発防止策を記録する。

---

## なぜ Google TTS 前提で進んだのか

- **品質優先**: Wavenet 音声は自然な英語発音を実現し、ユーザー体験目標（発音学習・会話の流れの把握）に合致していた。
- **既存の知見**: Google Cloud TTS は多くのプロジェクトで採用されており、導入判断は妥当だった。

---

## なぜ不可能な設計になったのか

### 1. ユーザージェスチャ境界の見落とし

- **設計**: 「タップ → API 非同期取得 → play()」の流れで、タップと play() の間に API 呼び出し（数百 ms〜数秒）が入る。
- **結果**: ブラウザのオートプレイ制約により、ユーザー操作の文脈が「非同期待ち」で切れ、`play()` がブロックされる。
- **現象**: 「会話全体を聞く」で 1 本目だけ Google TTS が流れ、2 本目以降はデバイス TTS にフォールバックする。または最初からフォールバック。

### 2. アーキテクチャ前提の誤認

- **初期設計**: Web/モバイルブラウザでの自動再生制約を「例外系」と扱い、アーキテクチャの前提として固定しなかった。
- **結果**: 「API キーを設定すれば動く」前提で進め、実際の再生フロー（タップ → 非同期 → play）がユーザージェスチャを維持できないことを早期に検証しなかった。

### 3. 連続再生要件との不整合

- **要件**: 会話学習画面で「会話全体を聞く」により、複数フレーズを連続再生したい。
- **制約**: 各フレーズごとに API 取得が必要であり、前の再生終了後に次の API を呼ぶ構造では、タップ直後の play() が維持できない。
- **結論**: 現行の「タップ → API → play」設計では、Web で連続的に Google TTS を通し続けることはほぼ不可能。

---

## 再発防止策

1. **音声機能の採用前**に、以下を PoC 合格基準として必須化する：
   - ユーザージェスチャ制約: タップ直後に play() を呼べるか（非同期を挟まないか）
   - 同期/非同期境界: API 取得と再生のタイミングがユーザー操作の文脈内で完結するか
   - 連続再生要件: 複数フレーズを想定する場合、各再生が上記を満たすか

2. **技術成立性レビュー**を仕様確定前に追加する：
   - プラットフォーム別（Web / iOS / Android）の自動再生・ユーザージェスチャ制約を事前に調査・確認する。
   - 「例外系」と決めつけず、アーキテクチャの前提として明文化する。

---

## 移行方針

- Google TTS 経路を廃止し、**OpenAI TTS（Supabase Edge Function 経由）+ flutter_tts フォールバック**の構成に切り替えた。
- API キーはクライアントに持たず、Edge Function 内で `OPENAI_API_KEY` を使用する。
- オートプレイ制約は OpenAI TTS でも同様に存在するが、サーバープロキシ構成により API 管理の負担を軽減しつつ、フォールバックで可用性を確保している。
