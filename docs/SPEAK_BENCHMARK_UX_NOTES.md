# Speak ベンチマーク UX ノート
Engrowth の Speak風UX原則（`SPEAK_STYLE_UX_PRINCIPLES.md`）を支える、「元ネタ」の観察メモ集。  
ここは**プロダクトの魂と気づき**を書く場所であり、実装指示は原則のほうに昇格させる。

---

## 0. このノートの役割

- Speak を実際に触って気づいたこと・記事から学んだことを、ラフに書き溜める。
- 「これは Engrowth の正史にしたい」と思えたものは、`SPEAK_STYLE_UX_PRINCIPLES.md` へ昇格させる。
- 技術的な詳細（Rive / viseme / 先読みなど）は `SPEAK_TECH_STACK_REFERENCE.md` に分離する。

---

## 1. コンセプト：インターフェースという「伴走者」

### 1.1 Cognitive Ease（認知の容易性）

- ユーザーが **「次にどこを見るべきか」一瞬も迷わない**。
- ビジュアルのポイント:
  - ホーム：学習パス（バブル・カード）で「今ここ」「次はこれ」「どこまで来たか」が一目で分かる。
  - 1画面1メッセージ：視線が迷う要素を極力排除。
- アニメーションの役割:
  - 次に注目してほしい要素が、**少し遅れて・目立つ動きで現れる**。
  - 視線誘導を「テキスト」ではなく「タイミング」と「動き」で行っている。

**Engrowthでの解釈**
- チュートリアルだけでなく、ホーム・学習画面でも「次アクション」を1つに絞り、**動きでそこへ誘導**する。
- 「3つのボタンが同じ重要度で並ぶ」レイアウトを避け、**今のステップに関係ない要素は極力黙らせる**。

---

### 1.2 Invisible UI（見えないインターフェース）

- メニューやタブなど「アプリっぽい UI」は奥に引っ込み、**学習体験そのものが画面の主役**になっている。
- チュートリアル・会話画面では、**フルスクリーンのコンテンツ＋最小限のコントロール**という構成が多い。
- 言語設定や課金など「事務的な要素」は、学習フローと視覚的に切り離されている。

**Engrowthでの解釈**
- 音声再生・会話・パターンスプリントなどの画面では、**上部バーや余計なアイコンを最小限に**。
- 「録音履歴」「設定」などは、学習中ではなく**前後のハブ画面**に寄せる。

---

### 1.3 Emotional Response（感情へのフィードバック）

- 正解・ミス・がんばりに対し、アプリが「温かい隣人」のように反応する。
  - 例: 柔らかい励ましメッセージ、控えめだが心地よいアニメーション。
- 大げさな花火よりも、「小さな褒め」を短いサイクルで積み重ねる印象。

**Engrowthでの解釈**
- 結果画面だけでなく、「1フレーズごとの小さな達成」にも **静かな祝福（軽いアニメ・色変化・短い文言）** を乗せる。
- 失敗時も「ダメ出し」ではなく、**次の一歩がすぐ分かるトーン**にする。

---

## 2. 継続を生む「仕組み」 — ゴルディロックスゾーンと可変報酬

### 2.1 Goldilocks Zone（難しすぎず・簡単すぎず）

- Speak の記事とレビューから：
  - 「マイクロフルエンシー（micro-fluency）」という概念で、**1ユニットごとに「これだけはスラスラ言える」状態**を作る。
  - 完全な文法正確性よりも、**現実に使えるブロック**を重視。
- 15〜30秒くらいで完結するミニ会話やフレーズ練習が多く、**「成功した感覚」が短い間隔で戻ってくる**。

**Engrowthでの解釈**
- パターンスプリントや30秒会話も、「このセットを通せばこのパターンだけは口から出る」という**小さなゴール**を明示する。
- チュートリアルでは「3分英会話」よりも、「30秒で1往復を回せる感覚」にフォーカス。

---

### 2.2 Variable Rewards（可変報酬）

- 完了時のメッセージや反応が、**毎回少しずつ違う**（レビューからの印象）。
- ストリークやSpeak Levelなど、**「次に何が解放されるか」** をほんのり匂わせる設計。

**Engrowthでの解釈**
- 完了メッセージやポップアップの文言を、**数パターン用意してローテーション**する。
- 同じボタンでも、日によって**小さな装飾や文言のニュアンスを変える**余地を残す。

---

### 2.3 Zero-Friction Progress（ゼロ摩擦の進行）

- 余韻のあと、自動で次のステップに進むことで、「もう一回やるか…」と考える時間を消している。
- 「Next」を押させることもあるが、**自動進行と手動進行がきれいにコンビネーション**されている印象。

**Engrowthでの解釈**
- チュートリアル・ガイドフローでは、**基本は自動進行＋必要なところだけ手動**。
- 自動進行の前後に必ず **1秒以上の「無音の余白」** を入れる（すでに原則に反映済み）。

## 3. Home / Progress UI に見る構造

### 3.1 Learn / Practice / Apply の三層構造

Speak 自身が打ち出している三本柱（記事より）:

- **Learn**: 音声・動画ベースでフレーズを学ぶ。
- **Practice**: ボキャブラリやユニット復習の「サイドクエスト」。
- **Apply**: ロールプレイやフリートークで実際に話す。

**UIとしてのポイント**

- ホーム画面がこの三層と直結しており、**「今日はどこをやるか」が構造で見える**。
- ユーザーは「やることを探す」のではなく、「用意された流れに乗る」体験になっている。

**Engrowthでの解釈**
- Engrowth でもホームを:
  - Learn: 30秒会話・3分会話などの「入力」
  - Practice: パターンスプリント・リピート系
  - Apply: 日々の報告・フリートーク
  に整理し、「どのカードがどの層か」が分かるデザインに寄せる。

---

### 3.2 Speak Level / ストリーク / 進捗パス

- Winter 2025 のアップデート:
  - 新しいホームは **「どこにいる」「何が次」「どれだけ進んだか」** を即座に見せることが目的。
  - Speak Level というフルエンシー指標を導入し、「話す力の成長」を1指標で見せようとしている。
  - ストリークには freeze / repair / milestone celebration があり、「切れたら即終了」ではない。

**Engrowthでの解釈**
- 「今日のミッション達成」「今週の音声提出数」など、**コンサル伴走とつながる指標**を1〜2本に絞って前面に出す。
- ストリークも、**「切らさないための優しさ」**（救済・やり直し）を前提に設計する。

---

## 4. まとめメモ（Engrowthへの翻訳の方向）

- Speak の強みは「マイクロフルエンシー」と「ゼロ摩擦の進行」にあり、  
  どちらも **「静寂と動きのタイミング制御」と「小さな成功体験の連打」** で支えられている。
- Engrowth では、
  - `SPEAK_STYLE_UX_PRINCIPLES.md` … 原則（どう動かすか）
  - このノート … 元ネタ（Speakで何が起きていると解釈するか）
  - `SPEAK_TECH_STACK_REFERENCE.md` … 技術（どう実装するか）
  という三層で管理する。
